## Summary

Build an interactive, fully client-side web demo under `playground/` where users
play with trained ACE models in the browser. A hand-rolled TypeScript port of
`ace.py`'s forward pass (parity-tested against the real PyTorch model) loads
exported weight blobs and runs instant amortized conditioning — no server. Two
demos: **GP-1D** (a playground: add/drag/delete points, infer kernel +
hyperparameters, pin latents and predict) and **Gaussian** (Beta-prior sliders +
observed `y`, with an analytic oracle overlay). The core nanoACE repo stays
torch-only; the playground is an explicitly non-core example.

## Scope

- **In scope**
  - `playground/` Vite + TypeScript project, no ML/UI framework (canvas/SVG only).
  - `export_weights.py`: checkpoint `.pt` → `manifest.json` + `weights.bin`.
  - Faithful TS forward port (embedder, separated ctx self-attn / tgt cross-attn,
    MDN + categorical heads, MDN/categorical density, mean/var) mirroring `ace.py`.
  - Parity test (`parity.py` → `parity.test.ts`) asserting TS == PyTorch raw head
    outputs within tolerance, covering every token path the demos use.
  - GP-1D demo: live predictive band, kernel-posterior bars, lengthscale/outputscale
    marginals; pin any subset of latents and predict; soft guardrails + OOD callout.
  - Gaussian demo: Beta-prior sliders (`mu`, `log_sigma`) + observed `y`; posterior
    marginals + posterior predictive; TS port of the analytic grid oracle overlay.
  - GitHub Pages deploy workflow; playground README; DEVLOG + README + AGENTS updates.
  - Commit the (small) exported weight blobs so the demo is self-contained.
- **Out of scope (v1)**
  - AR joint-sample draws ("draw sample functions") — deferred, easy to add later.
  - GP numerical oracle in-browser (live Cholesky-over-grid) — GP stays a playground.
  - WebGPU / quantization — plain fp32 typed arrays; model is tiny.
  - Retraining for multi-latent reveal — captured as a DEVLOG to-do, not done here.
  - Any change to `ace.py` / core training (the port is a read-only snapshot).

## Decisions locked (from discussion)

- In-browser TS port; GitHub Pages; both demos, GP first.
- GP = playground (no oracle). Gaussian = with analytic oracle overlay.
- "Pin a latent and predict" is the GP centerpiece.
- **Multi-pin (double/triple) is allowed** in the UI. It is **OOD** for current
  checkpoints (training reveals ≤1 latent at a time) and fires the OOD banner.
  A DEVLOG to-do records: retrain with a multi-latent reveal scheme to make it

- OOD: soft guardrails + explicit "outside training range" callout.
- Reuse existing `artifacts/*.pt`; ship weights as-is (no fp16).
- Lives in-repo at `playground/`.

### Phase 1: Weight export + TS forward + parity (the foundation)
**Goal**: TS forward reproduces the PyTorch model's raw head outputs within
tolerance for both checkpoints, across every token path. **No UI ships until green.**

**Work**:
- `export_weights.py` (run with project venv, `--task {gp1d,gaussian} --checkpoint
  <path> --out playground/public/models/<task>`):
  - Loads the checkpoint via the task's `load_checkpoint` to get a real `ACE`
    instance, so all derived constants come straight from `ace.py` (no divergence):
    `is_discrete, is_latent, has_bounds, bound_lo, bound_hi, cardinality,
    disc_offsets, max_cardinality`, plus `cfg` and head_dim = `d_model//n_heads`.
  - Emits `manifest.json` = `{task, cfg, derived, variables[], tensors[], dtype,
    byte_order}` where `variables[]` includes per-var name/kind/value_type/
    cardinality/transform/bounds + the derived per-var flags and `disc_offset`, and
    `tensors[]` is `{name, shape, offset, length}` over the state_dict in a fixed key
    order.
  - Emits `weights.bin` = concatenated float32 little-endian in that order.
  - Import detail: `export_weights.py` lives in `playground/` and adds the repo root
    to `sys.path` to import `ace` + the task module (no core changes). It iterates the
    full `state_dict` — note the bare `unknown` `nn.Parameter` is a top-level key (do
    **not** filter to only `*.weight`/`*.bias`), and the `persistent=False` buffers
    (`is_discrete`, …) are **not** in the state_dict, which is exactly why derived
    constants are read from the live `ACE` instance instead.
  - Provenance: committed blobs ← local checkpoint (gitignored `artifacts/*.pt`) ←
    training (`cfg` + `seed` recorded in the manifest), so the blobs are regenerable
    even though the checkpoint itself isn't committed.
- `src/ace/weights.ts`: decode `manifest.json` + `weights.bin` bytes into a tensor map
  (`{shape, data: Float32Array}` by name). **Separate the byte source from decoding** —
  a `fetch`-based loader for the browser and a raw-bytes entry point so `parity.test.ts`
  can feed Node `fs`-read bytes (vitest runs in Node, where `fetch` of a local file path
  is not available).
- `src/ace/schema.ts`: `Variable` type + `encodeValue`/`decodeValue` (affine
  `[-1,1]` for bounded continuous latents) mirroring `ace.py`.
- `src/ace/nn.ts`: `linear`, `layerNorm` (eps 1e-5), `gelu` (exact erf — accurate
  erf approx, ~1e-6), `softmax`/`logSoftmax`, `logsumexp`, `softplus`, and
  `multiHeadAttention` replicating `nn.MultiheadAttention` (packed `in_proj` split
  into q/k/v chunks, `1/sqrt(head_dim)` scaling, `key_padding_mask`, `out_proj`).
- `src/ace/model.ts`: `embed(tokens)` and `forward(context, target)` mirroring
  `ACE._embed` / `ACEBlock` / `ACE.forward`. **Parity-critical details to honor:**
  - payload select by mode: PRIOR → `value_embed(mean) + spread*spread_embed([mean,
    spread])` (spread_embed consumes the **full 2-vector**, value_embed only the mean);
    QUERY → learned `unknown`; else VALUE (discrete via
    `disc_value_embed[clamp(disc_offset + max(idx,0), 0, total_disc-1)]`, continuous via
    `value_embed`).
  - `value_embed` is **shared** between continuous values and the prior-mean location.
  - `mode_embed(mode.clamp(0,2))`; `x_embed` applied then **zeroed for latent tokens**;
    embedding `* mask`.
  - block: pre-LN, ctx self-attn → ctx MLP → tgt cross-attn (kv = `kv_ln(ctx)`) →
    tgt MLP; multiply ctx/tgt by their masks after each block; `final_norm` on tgt.
  - `forward` mirrors `ace.py`'s empty-context guard (errors / no-op if no active
    context token) so the TS path matches the UI guard.
- `src/ace/predictions.ts`: `contParams` (logSoftmax weights, loc, softplus+min_scale
  scale), `continuousMean`, `continuousVar`, MDN `logProb(y)`, categorical `logProb`
  (gather at `max(value_index,0)`)/`probs` with `_valid_logits` cardinality masking
  (fill invalid logits with float32-min as `ace.py` does, not `-Inf`; identical under
  softmax). **Note:** cardinality masking is *degenerate/untested* for both current
  checkpoints — the only discrete var (kernel, cardinality 4) equals `max_cardinality`,
  so no logits are ever masked. (Native decode helpers added only if a demo needs them;
  marginal plots normalize over a native grid so the affine Jacobian cancels — same
  trick as `diagnostics.normalized_moments`.)
- `parity.py` (project venv): build deterministic cases, run the real model, dump
  fixtures to `playground/test/fixtures/<task>.parity.json` (kept **out of** `public/`
  so they are not shipped to the site). Each case dumps `{name, context_tokens,
  target_tokens, embed_context, per_layer_ctx, per_layer_tgt, cont_raw, disc_logits}`
  plus a couple of derived `log_prob`/`mean`/`probs` values:
  - `cont_raw`/`disc_logits` are the raw head outputs (everything downstream derives
    from them).
  - `embed_context` = `_embed(context)` **before attention**, asserted directly so an
    embedder bug (PRIOR/discrete/x-zeroing) is localized rather than only inferred from
    target outputs (target outputs *do* transitively catch context-embedding bugs via
    cross-attention, but this pinpoints them).
  - `per_layer_ctx`/`per_layer_tgt` = `ctx`/`tgt` after each block, so a parity failure
    points at the first diverging layer instead of just the final logits.
  **Cases (cover every demo token path):**
  1. GP: data-only context + data targets + lengthscale/outputscale/kernel QUERY.
  2. GP: pinned kernel (discrete VALUE) + pinned lengthscale (zero-spread PRIOR) +
     remaining latent & data QUERY. **(Sole guardian of the discrete-VALUE-in-context
     and PRIOR-in-context embedding paths — must not be dropped.)**
  3. GP: a padded context (some masked tokens) to exercise `key_padding_mask`.
  4. Gaussian: finite-spread Beta PRIOR tokens (`mu`,`log_sigma`) + observed `y` +
     latent QUERY grid + new-`y` QUERY.
  5. Gaussian: one latent revealed as zero-spread PRIOR (reveal path).
  Coverage note: all seven token paths (data VALUE, finite PRIOR, zero-spread PRIOR,
  discrete VALUE, latent QUERY, data QUERY, padding) are touched; `_valid_logits`
  cardinality masking is *not* exercised (degenerate — see predictions.ts).
- `src/parity.test.ts` (vitest, Node): load manifest+bin (via the Node byte path) + the
  fixtures, run the TS forward, and assert with a combined relative+absolute criterion
  (see Verification) on `embed_context`, each layer's `ctx`/`tgt`, `cont_raw`,
  `disc_logits`, and the derived quantities, for every case and both models.

**Steps**:
1. Write `export_weights.py`; export both tasks → `public/models/`.
2. Implement `weights.ts`, `schema.ts`, `nn.ts`, `model.ts`, `predictions.ts`.
3. Write `parity.py`; generate `parity.json` for both tasks.
4. Write `parity.test.ts`; iterate on the TS ops until parity is green.

**Verification**:
- [ ] `npm test` parity green for both models, all 5 cases. Tolerance is a combined
      criterion `|ts−pt| ≤ atol + rtol·|pt|` (PyTorch float32 vs JS float64 makes flat
      1e-4-absolute unreachable after 4 layers): **raw outputs** `atol=1e-4, rtol=1e-3`;
      **derived** `continuousMean`/`continuousVar`/`logProb`/kernel `probs` `atol=1e-3,
      rtol=1e-3` (post-softmax/mixture, far more stable — and what users actually see).
      Don't chase bit-parity.
- [ ] `embed_context` and per-layer intermediates match within tolerance (localizes a
      failure to the first diverging op/layer).
- [ ] A spot-check script prints TS-vs-Python `cont_raw` for one case and matches.

### Phase 2: GP-1D demo
**Goal**: An interactive GP playground that foregrounds ACE's amortization and
latent inference, including multi-pin.

**Work**:
- `src/gp/demo.ts` — state → tokens → forward orchestration:
  - **Context** from UI state: data points (`var y`, x, value, VALUE); pinned kernel
    (discrete VALUE, `value_index=k`); pinned lengthscale/outputscale (zero-spread
    PRIOR, `prior=[encodeValue(native), 0]`).
  - **Targets**: dense x-grid for the predictive band (one forward → mean ± 2·√var);
    for each **unpinned** latent, a grid/label query (lengthscale & outputscale via
    encoded native grid → normalized marginal + mean/std; kernel via 4-label
    categorical → bar probs).
  - Pinned latents: skip query; draw a thick vertical line at the conditioned native
    value (kernel: lock/outline the chosen bar).
- `src/plot.ts` (shared, canvas) — drawing helpers: axes, line, ±2σ band, scatter,
  bars; used by both demos. GP view: context points, predictive mean + 2σ band, kernel
  bars, two latent marginals.
- **Interaction**: click to add a point, drag to move, shift/right-click to delete;
  pin controls = kernel selector (Unknown + 4 kernels) and lengthscale/outputscale
  sliders each with a pin toggle.
- **Guardrails / OOD**:
  - x hard-clamped to `[-1,1]` (function domain).
  - y draggable in a wider view (~`[-2.5,2.5]`); `[-1,1]` reads "normal", beyond it
    lightly shaded; a point flags OOD at `|y| > 2.0` (generous; training y tails
    exceed 1).
  - context count > 14 (`max_context`) → soft hint.
  - **≥2 latents pinned → OOD banner** (multi-pin allowed but OOD for now).
  - Banner names the reason(s): e.g. "2 points beyond training y-range • 3 latents
    pinned (not seen together in training)".
  - **Empty-context guard**: the model needs ≥1 active context token; if no points
    and no pins, disable prediction and prompt "add a point or pin a latent".
  - All thresholds and view bounds (`Y_OOD`, `MAX_CONTEXT_HINT`, y-view range, grid
    sizes) live as clearly-named constants in `src/config.ts`, not scattered as magic
    numbers — trivially tunable.

   matching fixed case (eyeball mean/RMSE-scale agreement).
2. Add interaction (add/drag/delete), then pin controls + thick-line/locked-bar.
3. Add guardrails + OOD banner logic.

## Risks & mitigations

- **Parity mismatch** (MHA chunking, exact-erf GELU, softplus, LN eps): caught by the
  parity test (embed + per-layer + raw + derived). Gate is a relative+absolute
  tolerance, **not** flat 1e-4 — PyTorch-float32 vs JS-float64 makes bit-parity
  unreachable; iterate on the ops, not on the tolerance floor. This is the linchpin gate.
- **Architecture drift** ("settled-ish" core): the port is a pinned snapshot; the
  parity test fails loudly and localizes any drift; DEVLOG records the snapshot.
- **Multi-pin OOD**: by design; surfaced via the banner; retrain tracked in DEVLOG.
- **5 MB GP blob**: one-time, cached; acceptable. fp16 deferred.
- **Empty context / all-masked keys** would error in the model: UI empty-context
  guard prevents it (Gaussian always has prior tokens; GP requires ≥1 point/pin), and
  the TS `forward` mirrors `ace.py`'s guard.
- **Periodic kernel period is fixed** (`period=1.0` over `[-1,1]`): note in UI/README;
  not a bug, a fixed generative choice.

## Resolved decisions (formerly open questions)

1. **Pages deploy**: manual `workflow_dispatch` only — no auto-deploy on push.
2. **OOD thresholds**: defaults accepted (GP `|y|>2.0`, Gaussian `|y|>3.0`, count `>14`)
   and centralized as clearly-named, tunable constants in `src/config.ts`.
3. **Plot tech**: canvas throughout via a shared `src/plot.ts` — one dependency-free
   render path, best for the live band redraw and fine for the static panels.
